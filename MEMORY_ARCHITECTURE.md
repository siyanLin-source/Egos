# MEMORY_ARCHITECTURE — TA 记忆系统架构白皮书（仓库版）

> **本文档定义「人生百科全书」的核心技术：TA 的记忆操作系统（Memory OS）。**
> 两个用途：① 所有 Codex / Claude Code 实现会话的架构母版；② 防跑偏锚点——任何新想法，先到第 6 节找到它的位置，再决定什么时候做。
> 与 PROJECT.md 的关系：本文档是其记忆系统章节的完整展开，与 PROJECT.md 并列存放于仓库根目录。

---

## 0. 一句话定义

TA 的记忆不是大模型自带的功能，而是我们在大模型之外建造的一套**记忆操作系统**：**写入 → 组织 → 检索 → 巩固 → 治理**五个环节构成完整生命周期，**L1–L4 四层记忆结构**决定任何时刻 TA"脑子里装着什么"。大模型是可更换的推理引擎；记忆数据、治理规则、以及随时间不断加厚的用户档案，全部独立于任何具体模型存在。

设计总纲一句话：**默认快、必要时深、离线时聪明。**

## 1. 模型策略：不训练大模型

### 1.1 大模型在系统中的角色

LLM 是**无状态的推理引擎**。每一次对话调用，我们临时向它注入：TA 人格系统提示词 + L1 核心画像 + L2 近期状态 + L3 检索结果 + 当前对话历史。它生成回复，然后什么都不记得。记忆 100% 存放在我们自己的数据库（Supabase）中。

由此推出三个结论：

1. **记忆归属于我们自己的数据层。**换模型不丢失任何记忆。
2. **模型进步的红利自动获得。**模型每年更聪明、更便宜，系统自动变好，无需重训任何东西。
3. **系统的长期价值在数据积累与治理规则的持续完善，不在绑定某个具体模型。**因此架构必须始终保持模型无关（model-agnostic）。

### 1.2 三个模型岗位

同一个系统里，不同任务用不同模型。这不是妥协，是正确设计。

| 岗位 | 职责 | 能力要求 | 当前选型 | 成本策略 |
|---|---|---|---|---|
| **对话模型**（TA 的嘴和情商） | 与用户实时对话 | 高情商中文表达；严格遵守人格规则（少提问、多在场）；能把记忆"织入"话语而非背诵 | Claude Sonnet | 用最好的，不省 |
| **提取模型**（后台书记员） | 对话结束后提取结构化信息：实体、情绪、类别、危机识别、profile facts | 格式稳定（JSON 不崩）> 聪明；危机识别不能漏 | Claude Haiku | 便宜量大 |
| **巩固模型**（夜间整理师） | 去重、冲突解决、显著性打分、叙事合成 | 长上下文、跨事件推理 | Sonnet 级 | 离线运行，走 Batch API（约半价） |
| **嵌入模型** | 向量化 | 中文语义质量 | text-embedding-3-small | 已是便宜项 |

### 1.3 换更笨的模型会发生什么（按岗位）

- **对话模型降级**：用户直接看得见。典型症状：无视注入的记忆；把记忆列清单式背出（"根据记录，你在 3 月 5 日……"——瞬间出戏）；违反一次一问规则；共情模板化。**这个岗位不能省。**
- **提取模型降级**：JSON 格式崩、实体漏提、危机误判。格式问题可用重试 + schema 校验兜底；**危机识别的准确率是底线，不达标不能换。**
- **巩固模型降级**：叙事质量下降。但用户不看过程只看结果，可接受中档模型 + 更强的提示词工程。

### 1.4 国内版模型策略（Phase 2）

「你和你」（国内市场）：Claude 不可直连且有合规要求 → 同一套记忆 OS 之上换 DeepSeek / Qwen / GLM 作为推理引擎。因为架构是模型无关的，切换 = 换 API 端点 + 重新调校人格提示词 + 小规模回归测试（用同一组测试对话对比新旧模型输出），**不是重建**。一句话：一套 schema，两地部署，两个可换引擎。

### 1.5 微调与自研的决策

- **现在不微调。**系统提示词 + few-shot 示例可达到人格一致性的 95%。
- **未来可选项**：积累几千段高质量对话后，对开源模型做轻量 LoRA 微调以固化 TA 语气——定位是**成本优化项**（让便宜模型达到贵模型的语气水准），不是能力项。排 Phase 3+。
- **自研预训练大模型：永远不做。**投入产出完全不成立。

## 2. 四层记忆结构（L1–L4）

这四层回答同一个问题：**任何一个时刻，TA 脑子里装着什么？**

### L1 核心画像层（语义记忆 · 常驻）

- **内容**：稳定身份事实（出生地、职业、家庭结构）、关键人物关系摘要、稳定偏好（爱吃辣、养猫）、当前人生主题（正在创业 / 备孕 / 换城市）。
- **来源**：`profile_facts` 中显著性得分 top-N 且稳定性等级 ≥ 缓变的事实。
- **Token 预算**：500–800。预算是硬约束——画像必须精炼，靠显著性分数竞争入场。
- **更新频率**：夜间巩固任务负责晋升 / 降级。
- **进入对话的方式**：每次对话**无条件注入** system prompt。零检索延迟，因为它不需要检索。
- **现状**：表已建成 ✓，注入逻辑待修复（当前 Sprint 3 blocker）。

### L2 近期状态层（工作记忆 · 常驻）

- **内容**：近 14 天 / 最多 8 条 entry 摘要；瞬时状态（上火、失眠、赶 deadline）。
- **Token 预算**：400–600。
- **更新频率**：实时，每次新对话拉取。
- **排除规则**：`is_crisis` 条目一律排除；近期强负面情绪条目默认排除（用户主动提起才接）。
- **现状**：已建成，真机验证 ✓。

### L3 情景记忆库（情景记忆 · 按需）

- **内容**：全部 entries——原文 + 摘要 + 向量 + 结构化字段（时间、人物、地点、情绪、类别）。这是**事实的最终来源**，append-only。
- **进入对话的方式**：反射式混合检索 top-k（k=3–5）注入；或 Agent 工具调用（见 3.3）。
- **现状**：存储与提取 ✓；**对话内检索 ✗（未开始，排在 L1 注入修复之后）**。

### L4 叙事洞察层（自传体记忆 · 按需 + 定期呈现）

- **内容**：跨事件合成的叙事。三种形态：
  1. **关系轨迹**："和妈妈：3 月因催婚紧张 → 5 月生日和解 → 6 月主动打了两次电话"。
  2. **人生线索**："换工作这个决定：2 月萌芽 → 3 月犹豫（钱 vs 意义）→ 5 月下定决心 → 情绪从焦虑转向踏实"。
  3. **模式识别**："每次大项目收尾后，你会低落一周左右——这是第三次了"。
- **来源**：夜间巩固 Agent 生成，存入 `narratives` 表（未来新表，字段建议：subject_type[person/theme]、subject_id、narrative_text、supporting_entry_ids、updated_at）。
- **进入方式**：a) 用户问跨时间问题时检索注入；b) 周报 / 年度报告主动呈现（对应 MVP 文档中的小结卡片）。
- **现状**：未开始，Phase 2。
- **设计意义**：这一层是"超越人类记忆"的具体所在。人类朋友只记得模糊印象、忘掉细节；本系统同时持有**原始细节 + 因果叙事 + 跨年模式**，三者人脑都做不到同时保真。

## 3. 五个生命周期环节（完整规格）

### 3.1 写入（Encoding）——双轨制是根基

每次交互产生两种性质完全不同的记忆，物理上分表、逻辑上分轨，绝不混淆：

**轨道一：entries（情景记忆）**
- Append-only 事件日志，**永不覆盖、永不修改**。
- 字段：时间戳、正文、摘要、人物、地点、情绪、类别、is_crisis。
- 提取时机：对话结束（新对话开启时 drain 上一段，含保守兜底 entry）——已建成 ✓。

**轨道二：profile_facts（语义事实）**
- 从情景中提炼的、关于"这个人是谁"的可修订知识。可更新、可被推翻、可降级。
- **每条事实必须携带四个元数据**：
  1. `provenance`：支撑该事实的 entry id 列表。作用：可溯源、可审计、用户质疑时可展示证据。
  2. `confidence`：0–1。初值来自提取模型判断，随后续证据累积上调、随矛盾证据下调。
  3. `mention_count`：被提及次数。这是权重系统的原材料（见 3.4 显著性公式）。
  4. `stability`：稳定性等级，四档——
     - **不可变**（出生地、母语）：一旦确认永久有效；
     - **稳定**（职业、家庭结构）：可被明确事件推翻（换工作、搬家）；
     - **缓变**（偏好、习惯）：随时间自然漂移，带新近度衰减；
     - **瞬时**（上火、失恋初期、赶 deadline）：**带 TTL（默认 7 天），到期自动失效，绝不进入永久画像。**

**写入纪律（提取模型的提示词级规则）：**
- 只提取"关于这个人是谁"的耐久信息；
- 顺带提及的路人不建档（selective extraction，已实现 ✓）；
- 情绪状态归 L2 瞬时轨道，永不写成 profile fact（既有原则，在此落为字段级规则）；
- 同一事实重复出现 → upsert 去重 + mention_count 自增（已有 upsert 机制 ✓）。

### 3.2 组织（Organization）——人是一等公民

散落的记忆要长成结构，核心是**实体图**：

- **person 节点（一等实体）**：每个人物挂载——别名集合（妈妈 = 母亲 = 老妈）、相关 entries、相关 facts、关系情感轨迹（由 L4 生成并持续更新）。已有 topics 表 + 人物列表（entry 计数）✓，person detail view 进行中 ◐。
- **地点 / 店铺 / 话题（二等实体）**：标签级组织即可，不建独立轨迹。
- **时间轴 + 类别轴**：已建成 ✓。
- **别名合并**：夜间任务基于同名 / 共现 / 上下文相似度提出合并建议；高置信自动合并，低置信留给用户一键确认（路线图第 2 项）。

"人际关系网络"的正确理解：不是一张社交图谱式的可视化，而是——**任何时刻问 TA"我和妈妈最近怎么样"，它能给出有时间纵深的回答。**这才是网络的意义。

### 3.3 检索（Retrieval）——精度大于召回

**路径 A：反射式检索（默认路径，每条用户消息执行）**

流水线（在 `app/api/chat/route.ts` 内实现）：

1. **并行执行**（Promise.all，互不等待）：
   - a) 用户消息嵌入（text-embedding-3-small，约 80ms）
   - b) L1 画像读取（直接查表，约 15ms）
   - c) L2 近况读取（直接查表，约 15ms）
   - d) 实体别名匹配（字典 / 正则，约 5ms）
   - e) 时间表达式解析（"去年十月" → date range，规则解析，约 5ms）
2. **pgvector 混合查询**：向量相似度 + 实体过滤 + 时间过滤 + is_crisis 排除。个人级数据量（数千条）下约 30–50ms。
3. **置信门槛**：top1 相似度低于阈值 → **不注入任何 L3 内容**。原则：宁可无记忆，不可错记忆。
4. **注入格式**：以摘要形式提供给对话模型，并附使用指引——"自然织入对话，勿背诵，勿罗列日期，勿说'根据记录'"。

**性能预算**：检索全程 p95 ≤ 200ms，之后即开始流式输出（Vercel AI SDK 已支持流式）。用户感知 ≈ 无感。这就是"默认快"的实现：**不是 Agent 决定查什么，而是流水线固定执行一次**——快是架构自带的，不靠优化。

**路径 B：Agent 式检索（升级路径，条件触发）**

- **触发条件（确定性规则判断，不靠模型自觉）**：
  1. 显式回忆动词："还记得 / 上次 / 什么时候 / 我去年 / 我有没有说过"；
  2. 歧义信号：反射检索返回 ≥2 个强候选且指向不同事件；
  3. 跨时间推理："这两年 / 变化 / 一直以来 / 越来越"。
- **实现**：对话模型获得一个 `search_memory` 工具（参数：query、person、date_range、category、emotion），**限 2–3 次调用**封顶，防止延迟失控。
- **UX 掩护**：触发时 TA 先说一句"我翻了翻我们之前聊的……"。延迟被转化为拟人化特征——朋友翻日记本来就要几秒钟。
- **性能预算**：p95 3–5 秒。可接受，因为用户问的是回忆题，心理预期本来就是"让它想一想"。

**歧义澄清机制**：当多个强候选指向不同事件时，用掉人格规则里唯一的提问配额："你说的是过年那次，还是上个月那次？"——**澄清本身就是"它真的记得"的证明**，是加分动作而非降级体验。

**路径 C 见 3.4（离线，无延迟约束）。**

### 3.4 巩固（Consolidation）——夜间 Agent

**运行机制**：
- 每日凌晨定时触发：Vercel Cron 或 Supabase pg_cron → Edge Function。
- **水位线机制**：只处理上次运行之后的新增数据，幂等、可重跑、失败可续。
- 走 Batch API（约半价），离线任务对延迟零要求，成本压到最低。

**四个 Job（按执行顺序）：**

**Job 1 去重与冲突解决**
新事实与旧事实矛盾时（"搬到上海了" vs "住在重庆"）：旧事实**不删除**，打上 `superseded_at` 时间戳降级归档——不再进入任何注入面，但历史可查。这一条机制就是"前任问题"（分手后 TA 还提前任 = 一次卸载）的系统级解法。

**Job 2 显著性打分（Salience Scoring）——权重系统的心脏**

```
salience = w1 · log(1 + mention_count)        // 频次：提了 8 次的火锅店
         + w2 · emotion_intensity              // 情绪强度：大哭一场 > 随口一提
         + w3 · recency_decay(t)               // 新近度衰减：偏好半衰期约 90 天，瞬时状态更短
         + w4 · user_confirmation               // 确认信号：TA 提起后用户接话展开 = 强化；没接 = 降权
         + w5 · category_weight                 // 类别权重：人物关系 > 人生主题 > 饮食偏好
```

- **晋升规则**：各类别按 salience 取 top-N，且总量不超过 L1 的 token 预算 → 晋升常驻画像；衰减跌破下线 → 降回 L3 待检索。
- **关键设计表述**：画像不是人工挑选的，是分数自己浮上来的。系统每晚重新回答一次"关于这个人，最重要的是什么"。

**Job 3 叙事合成（L4 生成）**
以 person / 人生主题为单位，读取相关 entries 的时间序列，生成或增量更新叙事文档（关系轨迹、决策过程、情绪模式）。写入 `narratives` 表。

**Job 4 生命周期维护**
瞬时状态 TTL 过期清理；别名合并建议生成；低置信垃圾事实清洗；孤儿数据修复。

**一个帮助理解的类比**：
> 就像人类睡眠时海马体把白天的经历转存进大脑皮层，TA 每晚都会"睡一觉"——把白天聊过的一切，整理成对这个人更深的理解。

### 3.5 治理（Governance）——知道 ≠ 说出来

记忆的**使用**受社交得体规则约束，而不是"检索到了就说"。这一层是产品人格与安全规则的合流点。

**规则集：**
1. `is_crisis` 条目：不进任何注入面、不进任何展示面（规则已定 ✓；**迁移必须在 Supabase 确认已应用**，git push ≠ migration applied）。
2. 近期强负面情绪：默认不主动提起，用户先开口才接。
3. 敏感类别（健康、亲密关系冲突、财务）：只在用户已进入同话题的语境下使用。
4. 开屏问候：只从**稳定 + 中性/积极**事实中取材（route.ts fix brief 的范围）。
5. **记忆透明度（用户侧治理）**：「关于你」页面就是用户可见的记忆账本；未来支持用户对单条 fact 纠错 / 删除，纠错信号回流 confidence 字段。用户能看见并修正 TA 记得什么——这既是信任设计，也是数据质量的免费清洗渠道。

**产品哲学一句话**：
> TA 像一位有分寸的老朋友——很多事它知道，但它懂得什么时候不说。

## 4. 端到端时序（两条时间线）

**白天：一条消息的旅程**
1. 用户发送："推荐我今天去哪吃饭？"
2. 并行：嵌入 + L1 读取 + L2 读取 + 实体匹配 + 时间解析（~100ms）
3. pgvector 混合检索 top-k，过置信门槛（~50ms）
4. 组装上下文：人格提示词 + L1（她爱吃辣、常去老地方火锅）+ L2（三天前刚吃过火锅；本周上火）+ L3（上个月提过想试那家云南菜）
5. 对话模型流式输出："按你的心头好该推老地方，不过你前几天刚去过，而且这两天上着火——要不要试试你上个月说想去没去成的那家云南菜？"
6. 对话结束后：Haiku 提取本段 → entries + facts 更新，mention_count 自增。

**注意第 5 步的分层**：记住偏好是记忆（L1），知道刚吃过、知道上火是近况（L2），想起云南菜是检索（L3），**综合四者给出"换口味"的建议是在记忆之上推理**——最后这半步才是"超越人类记忆"的具体含义。

**夜间：巩固流水线**
1. 凌晨 3:00 Cron 触发 → 读取水位线
2. Job 1 去重冲突 → Job 2 显著性重算 + L1 晋升/降级 → Job 3 叙事更新 → Job 4 生命周期维护
3. 更新水位线，记录运行日志（成功/失败/处理条数）
4. 用户第二天醒来，TA 对这个人的理解比昨天深了一层——用户无感知，但周报会呈现。

## 5. 用户体验设计原则（记忆如何被"感觉到"）

1. **织入而非背诵**：记忆以从句形式出现——"上次你说那家汤底有点咸，这次要不要换一家？"绝不出现"根据我的记录显示"。这条写进对话模型的注入指引。
2. **澄清即证明**：歧义时的那一个问题，是记忆真实性的展示窗口。
3. **延迟拟人化**："我翻了翻我们之前聊的……"——把 Agent 检索的 3 秒变成老朋友翻日记的画面感。
4. **记忆的回报感**：周报 / 小结卡片让用户看见"记录正在长成什么"（情绪曲线、关键词、TA 的一段话）。记录是有回报的，回报要可见。
5. **错误记忆零容忍**：置信门槛 + 宁缺勿错 + 用户纠错通道。张冠李戴一次，信任崩塌一次。
6. **分寸感即人格**：治理层的每条规则，用户体验到的是"它懂事"。

## 7. 风险与红线

| 风险 | 后果 | 防线 |
|---|---|---|
| 错误记忆（张冠李戴） | 信任一次性崩塌 | 置信门槛、宁缺勿错、歧义必澄清、用户纠错通道 |
| 过期事实（"前任问题"） | 一次卸载 | supersede 机制、稳定性分级 |
| 瞬时状态永久化（"你不是不能吃辣吗"） | 出戏、显得不懂人 | stability 四级 + TTL |
| 危机内容进入注入/展示面 | 对脆弱用户造成伤害 | is_crisis 全面过滤（迁移须确认）、治理层规则 1–4 |
| 隐私 | 信任根基 | RLS ✓、生物锁、伪装模式、对用户诚实披露"AI 处理时可见"（不过度承诺）、API key 永不进客户端 |
| 成本失控 | 运营成本压力 | 三岗位分级用模型、Batch API、嵌入选便宜档、L1/L2 token 硬预算 |
| 延迟失控 | 陪伴感碎裂 | 反射路径 p95 ≤200ms、Agent 路径限 2–3 次调用、巩固全部离线 |
